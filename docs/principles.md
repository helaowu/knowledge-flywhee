# 设计原理

## 复利 vs 自动化

这个区分是整个系统的根基。

**自动化**：一次性投入，固定收益。
- 例：写一个周报脚本，以后每周自动跑
- 好处：省时间
- 局限：脚本本身不会变好。第 100 次跑和第 1 次跑出来的东西一模一样

**复利**：每次使用，收益递增。
- 例：每次查数后把新表的信息记下来
- 好处：下次碰到这张表不用重新学
- 关键区别：第 100 次使用时，系统比第 1 次使用时更强

## 为什么 AI 需要这套东西

AI（LLM）的几个结构性缺陷：

1. **无状态**：每次对话从零开始，没有跨会话记忆
2. **会遗忘**：上下文窗口有限，长对话会压缩/截断
3. **不反思**：不会主动从错误中提取规律
4. **不积累**：做过一次复杂任务，下次不会更快

这套系统在 AI 外部搭建了"记忆"和"学习"能力——用文件系统弥补模型的无状态缺陷。

## 三个核心循环

### 循环 1: 错误 → 规则

```
犯错 → 被纠正 → 记入 raw_log → 同类出现 3 次 → 写入正式规则 → 永不再犯
```

关键：不是"下次注意"，是"系统层面杜绝"。3 次阈值防止把一次性边界情况当成规则。

### 循环 2: 使用 → 知识

```
查了一张表 → 记下表结构/坑/口径 → 下次碰到直接有底 → 不用重新学
```

关键：自动触发，不靠人提醒。每次任务完成后检查是否有新东西值得记。

### 循环 3: 压缩 → 保留

```
对话太长要压缩 → 压缩前自动提取关键信息存入文件 → 压缩后不丢记忆
```

关键：解决 LLM 最大的结构性问题——上下文窗口有限。

## 升级阈值：为什么是 3 次

- 1 次：可能是边界情况，不值得写规则
- 2 次：开始有模式，但可能巧合
- 3 次：确认是系统性问题，必须修

太低会产生大量噪音规则。太高会让同一个错反复犯。3 是实践中找到的平衡点。

## 和 RAG / Fine-tuning 的区别

| 方法 | 优点 | 局限 |
|------|------|------|
| RAG | 能检索大量文档 | 被动检索，不主动学习 |
| Fine-tuning | 永久改变模型行为 | 贵、慢、不可逆、难以针对个人 |
| 这套系统 | 轻量、即时、可逆、个人化 | 依赖提示词遵守能力 |

这套系统本质上是"结构化的 prompt 管理"——但它不是一次性写好的 prompt，而是一个会自我进化的 prompt 系统。
